# 在 act4 框架下添加新非特权指令测试套件（Zibi: `beqi` / `bnei`）

ACT4 是一个用于 RISC-V 指令验证的自动化测试框架，负责测试用例的生成、执行以及结果验证，从而系统性地保证指令实现的正确性。
在本次工作中，我们的目标是在 ACT4 框架下，为实验性扩展 Zibi（包含两条新指令 BEQI 和 BNEI）添加测试套件。具体来说，我们需要基于 ACT4 的测试流程，完成指令测试的生成与执行，并结合 Sail 编写的RISC-V 参考模型生成对应的执行结果（signature）。
最终目标是构建一套带有 Sail 参考结果的自检测试（self-checking tests），使其能够在 DUT（Design Under Test）上运行，并通过结果比对验证DUT实现是否符合规范。

---

## 测试添加流程

在 act4 里，为一个新扩展（例如 `Zibi`）接入测试，通常需要进行以下7个步骤：

1. `testplans/<EXT_name>.csv`：声明“指令名、指令类型、开什么覆盖点”；
2. `sample_<TYPE>.sv`：定义新类型指令如何从 trace 里取字段并采样；
3. `formatters/types/<type>_type.py`：定义该类型的指令如何拼接成测试汇编；
4. `disassemble.svh`：让反汇编能正确显示新指令；
5. `coverpoints/*.py + templates/*.sv`：实现覆盖点和交叉覆盖；
6. `make ... tests`：生成目标目录下的测试汇编文件。
7. `CONFIG_FILES=<your_config_directory>/test_config.yaml make --jobs $(nproc)` 生成自检测室。

## 1. Step1：在 `testplans` 里定义测试计划

非特权指令测试的生成由一个 CSV 格式的测试计划驱动，该测试计划定义了扩展中所有指令，以及适用于每条指令的覆盖点。每个扩展都应在 testplans 目录下包含一个名为 <extension_name>.csv 的测试计划文件。

### 1.1 文件示例

新增：`testplans/Zibi.csv`

|Instruction|Type|RV32|RV64|cp_asm_count|cp_rs1|cp_rs1_edges|cr_rs1_cimm_edges|cp_offset|
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
beqi|BI|x|x|x|x|x|offset|x|
bnei|BI|x|x|x|x|x|offset|x

### 1.2 各字段含义

- `Instruction`：指令名（`beqi` / `bnei`）
- `Type`：指令类型（这里是新增的 `BI`）
- `RV32/RV64`：是否在 32/64 位场景启用
- `cp_* / cr_*`：打开哪些覆盖点（point / cross）
-`X` :  表示启用覆盖点。另有特殊的覆盖点版本，需要用对应字符串指明，如 `offset`

## 2. Step2：新增指令类型 BI（模板 + 格式化器）

> 因为 `beqi/bnei` 同时包含 `rs1 + cimm + branch offset`，无法完全复用传统 `B` 类型，所以新增 `BI` 指令类型， 或说添加新的指令格式。添加一种新的指令格式需要新增一个 SystemVerilog 的采样模板以及一个 Python 的指令格式化器。

### 2.1 覆盖采样模板：`sample_BI.sv`

文件：`generators/coverage/templates/sample_BI.sv`

```systemverilog
// sample_BI.sv（示意）
// 作用：告诉覆盖框架，BI 类型指令需要采样哪些字段

BI: begin
    // 第0个操作数：源寄存器 rs1
    ins.add_rs1(0);

    // 第1个操作数：5-bit 立即数 cimm
    // add_cimm 会把操作数字段解析并写入 ins.current.cimm
    ins.add_cimm(1);

    // 第2个操作数：分支偏移 imm
    ins.add_imm_addr(2);
end
```

- INSTR 会被替换为指令名称，并作为 case 语句中的键。  
- ins 是一个数据结构，用于保存当前指令的所有信息。  
- 各种 add_* 函数用于将指令汇编字符串中的参数赋值给变量。数字表示从汇编字符串中取第几个参数赋值给指定变量。例如，在上述代码中，第一个参数赋值给 rs1，第二个赋值给 cimm，第三个赋值给 offset。

#### 注意事项：
- 模板要和 `testplans` 中 `Type=BI` 对上；
- 采样顺序要与指令语义一致（`rs1, cimm, offset`）；
- 采样字段名要和 trace 数据结构匹配（例如trace中要有 `cimm`这个字段）。

### 2.2 新字段接入 - 以`cimm`为例

如果框架里原本没有该字段，需要两步将其添加到框架中：

1. 在 `framework/src/act/fcov/coverage/RISCV_trace_data.svh` 中新增字段（例如 `cimm`）；
2. 在 `framework/src/act/fcov/coverage/RISCV_instruction_base.svh` 中实现 `add_cimm(...)`，用于从操作数字段取值并写入 `current.cimm`。

示意代码：

```systemverilog
// RISCV_instruction_base.svh（示意）
virtual function void add_cimm(int offset);
    // 从当前指令第 offset 个操作数读取立即数，并保存到 cimm
    current.cimm = get_imm(ops[offset].key);
endfunction
```

### 2.3 指令格式化器：`bi_type.py`

标准的覆盖点生成器依赖指令格式化器来生成测试每条指令所需的汇编代码。每种指令类型都需要一个 Python 生成器，用于生成对应的汇编测试代码。  
以下内容适用于所有指令格式化器：
- 所有指令格式化器必须放在 generators/testgen/src/testgen/formatters/types 目录下。该目录中的所有 Python 文件都会被自动发现并导入。
- 所有指令格式化器函数都必须使用 @add_instruction_formatter("<TYPE_NAME>", <type_name>_config) 装饰器。这用于告诉框架该生成器适用于哪种指令类型，以及如何为其生成参数。
- <type_name>_config 参数是一个 InstructionTypeConfig 对象，其中包含该指令类型所需的参数（required_params）以及这些参数的约束（例如寄存器范围 reg_range、立即数范围 imm_range 等）。更多细节可参考 `generators/testgen/src/testgen/formatters/registry.py` 中的 InstructionTypeConfig 文档说明。
- 所有指令格式化器函数必须使用如下函数签名：
```Python
def format_name_type(instr_name: str, test_data: TestData, params: InstructionParams) -> tuple[list[str], list[str], list[str]]:
```
  - instr_name 是当前正在测试的指令名称，这使得同一类型的格式化器可以复用于多条指令。
  - test_data 是一个 dataclass，在整个测试生成流程中传递，用于存储 signature 计数、测试值、调试信息等。
  - params 是一个 dataclass，包含该指令所有参数的取值（如 rs1、rs1val、immval 等）。完整定义可见 `generators/testgen/src/testgen/data/params.py`。
  - 该生成器必须返回一个包含三个字符串列表的元组：
    - 用于初始化测试环境的代码
    - 测试本身（通常就是待测试的指令）
    - 用于检查测试结果的代码（通常是 signature 检查）

我添加了文件：`generators/testgen/src/testgen/formatters/types/bi_type.py`

#### 代码实例

```python
from testgen.asm.helpers import load_int_reg, write_sigupd
from testgen.data.params import InstructionParams
from testgen.data.state import TestData
from testgen.formatters.registry import InstructionTypeConfig, add_instruction_formatter

# BI 类型需要的最小参数集合：
# rs1/rs1val: 源寄存器及其值
# immval: cimm 立即数值
# temp_reg/temp_val: 用于分支结果检查的临时寄存器
bi_config = InstructionTypeConfig(
    required_params={"rs1", "rs1val", "immval", "temp_reg", "temp_val"},
    imm_bits=5,
    imm_signed=False,
)

@add_instruction_formatter("BI", bi_config)
def format_bi_type(instr_name: str, test_data: TestData, params: InstructionParams):
    # 断言参数完整，避免拼装出非法汇编
    assert params.rs1 is not None and params.rs1val is not None
    assert params.immval is not None
    assert params.temp_reg is not None and params.temp_val is not None

    # setup：初始化 rs1 和临时寄存器
    setup = [
        load_int_reg("rs1", params.rs1, params.rs1val, test_data),
        load_int_reg("branch taken value", params.temp_reg, params.temp_val, test_data),
    ]

    # test：生成被测指令，offset 指向本地标签 1f
    test = [
        f"{instr_name} x{params.rs1}, {params.immval}, 1f # perform operation",
    ]

    # check：若分支未跳转，会执行 LI(...,-1)；跳转后统一写签名
    check = [
        f"LI(x{params.temp_reg}, -1) # branch not taken, set temp_reg to -1",
        "1:",
        write_sigupd(params.temp_reg, test_data),
    ]

    return (setup, test, check)
```

## 3. Step3：在反汇编中支持新指令

非特权指令在 disassemble.svh 中进行解码。所有新增指令都需要添加到对应的 case 语句中。disassemble.svh 的作用是将指令编码转换为指令助记符（mnemonic）及其参数。具体的编码信息来自自动生成的头文件 RISCV_imported_decode_pkg.svh。该头文件由 riscv-opcodes 生成，不应手动修改。

修改文件：`framework/src/act/fcov/disassemble.svh`

```systemverilog
// 先把 5-bit cimm 转成展示值（这里实现了 0 -> -1 的约定）
automatic int cimm5 = (instr[24:20] == 0) ? -1 : instr[24:20];

case (decoded_instr)
    BEQI: $sformat(decoded, "beqi %s, %0d, %0d", rs1, cimm5, immBType);
    BNEI: $sformat(decoded, "bnei %s, %0d, %0d", rs1, cimm5, immBType);
endcase
```

## 4. Step4：新增覆盖点模板 + 覆盖点生成器

添加一个新的覆盖点需要同时为该覆盖点本身添加一个模板，以及一个用于生成该覆盖点测试的 Python 生成器。

### 4.1 覆盖模板实例（SV）

####  `cr_rs1_cimm_edges_offset.sv`

```systemverilog
// 交叉覆盖：rs1 边界值 × cimm 边界值 × offset方向
cr_rs1_cimm_edges_offset : cross cp_rs1_edges, cp_imm_edges_5bit_u_n0, cp_offset iff (ins.trap == 0) {
    
}
```

### 4.2 覆盖点生成器参数含义

以 `cr_rs1_cimm_edges_offset.py` 为例：

```python
@add_coverpoint_generator("cr_rs1_cimm_edges_offset")
def make_cr_rs1_cimm_edges_offset(
    instr_name: str,
    instr_type: str,
    coverpoint: str,
    test_data: TestData,
) -> list[str]:
```

- `instr_name`：当前指令名（`beqi` / `bnei`），用于生成具体汇编；
- `instr_type`：指令类型（`BI`），用于参数生成器选型；
- `coverpoint`：当前覆盖点名（用于 testcase 名、注释、组织输出）；
- `test_data`：生成上下文（寄存器池、XLEN、签名写入工具、随机态）；
- 返回 `list[str]`：最终输出到 `.S` 测试文件的汇编行列表。

### 4.3 覆盖点生成器代码实例

```python
from testgen.asm.helpers import load_int_reg, return_test_regs, write_sigupd
from testgen.coverpoints.registry import add_coverpoint_generator
from testgen.data.edges import IMMEDIATE_EDGES, get_general_edges
from testgen.formatters.params import generate_random_params

@add_coverpoint_generator("cr_rs1_cimm_edges_offset")
def make_cr_rs1_cimm_edges_offset(instr_name, instr_type, coverpoint, test_data):
    # rs1 取通用边界值（与 xlen 相关）
    rs1_edges = get_general_edges(test_data.xlen)

    # cimm 取 5-bit 边界，排除 0，并显式加入 -1
    cimm_edges = (-1,) + IMMEDIATE_EDGES.imm_5bit[1:]

    test_lines = []

    # 交叉组合：每个 rs1 边界 × 每个 cimm 边界
    for edge_val1 in rs1_edges:
        for imm_val in cimm_edges:
            # 为当前样本申请随机寄存器，并固定 rs1val/immval
            params = generate_random_params(
                test_data,
                instr_type,
                exclude_regs=[0],
                rs1val=edge_val1,
                immval=imm_val,
            )

            # testcase 标识便于覆盖回溯
            test_lines.extend([
                test_data.add_testcase(
                    f"rs1_{test_data.xlen_format_str.format(edge_val1)}_cimm_{imm_val}",
                    coverpoint,
                ),
                # 先设置 rs1
                load_int_reg("rs1", params.rs1, params.rs1val, test_data),
                # 通过前跳 + 后跳，构造 taken/not-taken 两条路径
                "0:",
                f"{instr_name} x{params.rs1}, {params.immval}, 3f",
                "1:",
                f"{instr_name} x{params.rs1}, {params.immval}, 0b",
                write_sigupd(0, test_data),
                "j 4f",
                "2:",
                f"LI(x{params.rs1}, 1)",
                write_sigupd(params.rs1, test_data),
                "j 4f",
                "3:",
                f"{instr_name} x{params.rs1}, {params.immval}, 2b",
                "4: done the test",
            ])

            # 释放寄存器，避免后续样本冲突
            return_test_regs(test_data, params)

    return test_lines
```

## 5. Step5：生成测试
执行命令

```bash
make clean-tests
make --jobs $(nproc) EXTENSIONS=Zibi tests
```

成功后会在测试树下生成扩展目录，常见为：

- `tests/rv32i/Zibi/`
- （如果启用 RV64）`tests/rv64i/Zibi/`

## 6. 生成自检测试
执行命令：
```bash
CONFIG_FILES=<your_config_directory>/test_config.yaml make --jobs $(nproc)
```
可以参考README.md 文件，了解配置文件如何设置。本例中，我们需要在框架配置(*.yaml)中添加Zibi扩展，并指明其版本。生成结果将位于`work/<cvw-*>/elfs/rv32i/zibi`中。
注意： 如果扩展还没有在UDB仓库中声明，则需要手动生成汇编代码和自检测试。
