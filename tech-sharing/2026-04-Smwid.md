# SmWorld / Smwid 技术分享

> 从“为什么需要 World 隔离”讲到“当前分支代码具体改了什么”

---

## 先从一个问题开始
通过这个问题，我们可以初步了解设计world的目的。
一台 SoC 上可能同时跑很多进程：

- 普通操作系统
- 安全监控程序
- 固件
- 设备 DMA
- 调试或管理代码

它们都会访问同一套物理地址空间。

问题是：谁可以访问哪段内存、哪个外设，应该由谁来决定？

---

## 只靠虚拟内存够不够？

虚拟内存主要是 CPU 内部看到的地址转换机制。

但 Worlds 要解决的是更底层的问题：

- 访问最终都会落到系统物理地址
- 外设和 DMA 也可能发起物理地址访问
- 总线、cache、memory controller、device 也要知道访问来自哪个隔离域

所以 Worlds 的关注点不是“虚拟地址怎么翻译”，而是“物理事务属于哪个隔离域”。

---

## 关键词：事务 transaction

这里的 transaction 可以理解成一次系统层面的访问请求。

例如：

- CPU 取一条指令
- CPU load 一段内存
- CPU store 一段内存
- 页表遍历读取一个 PTE
- 访问 CLINT / HTIF 这样的 MMIO 设备

事务里至少会有：

- 要访问的物理地址
- 是读、写还是取指
- 访问大小
- 现在还要加上：这个事务属于哪个 World

---

## 关键词：World

World 可以理解成一个“系统级隔离域”。

它不是一个进程，也不是一个虚拟地址空间，而是平台对物理资源做隔离时使用的分区。

一个 World 里可能有：

- 一个或多个 hart 上的执行上下文
- 某些内存区域
- 某些外设
- 某些可共享资源

World 的目标是：即使访问已经变成物理地址，也还能判断这次访问是否允许。

---

## 关键词：WID

WID 是 World Identifier，World 的编号。

可以把它理解成事务上的“隔离域标签”。

例如：

- WID 0：默认固件 World
- WID 1：普通操作系统 World
- WID 2：某个安全服务 World

当一个 hart 发出访问时，事务不仅带地址，也带 WID。

平台上的检查逻辑看到“地址 + WID + 访问类型”，才能判断是否放行。

---

## Worlds 的核心想法

以前的访问检查可能只看：

```text
物理地址 + 读/写/取指类型
```

Worlds 增加一项：

```text
物理地址 + 读/写/取指类型 + WID
```

这样同一个物理地址可以对不同 World 有不同权限。

例子：

- WID 0 可以读写启动固件区
- WID 1 只能执行普通内存，不能写固件区
- 某个共享 buffer 可以同时允许 WID 0 和 WID 1 访问

---

## 没有 Smwid 时的worlds扩展

如果平台支持 Worlds，但没有启用 Smwid：

- hart 仍然可以有一个固定 WID
- 这个 WID 由平台决定
- 规范里把它叫做 `pmwid`
- 软件不一定能通过 ISA 修改或读取它

---

## Smwid 增加的最小能力

Smwid 增加一个 M-mode CSR：

```text
mwid
```

它的作用是：

- 保存当前 hart 发出事务时使用的 WID
- 让 early boot 阶段的 M-mode 软件可以设置这个 WID
- 设置完成后可以锁住，防止运行期再改变

这就是 Smwid 的核心：给 hart 一个可锁定的 World ID 控制寄存器。

---

##  lock bit 的意义

Worlds 的安全模型里，早期启动代码更可信。

启动阶段可以做这些事：

- 设置当前 hart 属于哪个 World
- 设置平台资源权限
- 设置完成后锁定

锁定之后，运行期 M-mode 不能随便改变自己的 World。

这避免了“运行期代码把自己切回高权限 World”的风险。

---

## `mwid` 的字段结构

`mwid` 是 XLEN 宽。

XLEN 是当前 RISC-V 机器字长：

- RV32 上 XLEN = 32
- RV64 上 XLEN = 64

`mwid` 里有两个重要部分：

- 低位保存 WID
- 最高位 `XLEN-1` 是 lock bit

中间没有用到的位会被规整为 0。

---

## WID 字段宽度计算

平台支持多少个 World，由 `NWorlds` 决定。

如果只支持 2 个 World，只需要 1 bit：

```text
0 或 1
```

如果支持 4 个 World，需要 2 bit：

```text
0, 1, 2, 3
```

所以 WID 字段宽度是：

```text
ceil(log2(NWorlds))
```

---

## 现在回到代码：新增主文件

当前分支新增：

```text
model/extensions/Smwid/smwid.sail
```

它是 Smwid 实现的核心文件。

主要负责：

- 从 config 读取 `nworlds`、`pmwid`、`pmwidlist`
- 定义 world region 权限结构
- 实现 `mwid` CSR
- 计算当前访问使用哪个 WID
- 检查某次物理访问是否被当前 WID 允许

---

## 代码：读取平台配置

```sail
let sys_smwid_nworlds : int =
  config extensions.Smwid.nworlds

let sys_smwid_pmwid : xlenbits =
  trunc(config extensions.Smwid.pmwid : bits(64))

let sys_smwid_pmwidlist : xlenbits =
  trunc(config extensions.Smwid.pmwidlist : bits(64))

let sys_smwid_locked_on_reset : bool =
  config extensions.Smwid.mwid_locked_on_reset : bool
```

这几行把 JSON 配置变成 Sail 模型内部可用的常量。

---

## 配置项解释：`nworlds`

`nworlds` 表示平台最多支持多少个 World。

例如：

- `nworlds = 1`：只有一个 World，WID 字段宽度为 0
- `nworlds = 2`：两个 World，WID 字段宽度为 1
- `nworlds = 8`：八个 World，WID 字段宽度为 3

这个值会影响：

- `mwid` 里低几位是 WID
- mask 中哪些 bit 是合法的
- 配置校验允许哪些 WID

---

## 配置项解释：`pmwid`

`pmwid` 是 platform machine WID。

它表示平台给 hart 指定的默认 World。

reset 后，`mwid` 会从 `pmwid` 初始化。

可以把它理解成：

```text
系统启动时，这个 hart 默认属于哪个 World
```

---

## 配置项解释：`pmwidlist`

`pmwidlist` 是一个 bit mask。

bit i 表示 WID i 是否允许写入 `mwid`。

例子：

- `0x1`：只允许 WID 0
- `0x3`：允许 WID 0 和 WID 1
- `0x5`：允许 WID 0 和 WID 2

规范要求：`pmwid` 必须包含在 `pmwidlist` 中。

---

## 代码：计算 WID 字段宽度

```sail
function smwid_wid_width_for_nworlds(nworlds : int) -> int = {
  var width : int = 0;
  var representable_world_count : int = 1;
  while representable_world_count < nworlds do {
    width = width + 1;
    representable_world_count = representable_world_count * 2;
  };
  width
}
```

这段代码做的事就是不断扩大可表示数量，直到能覆盖 `nworlds`。

---

## 代码：`mwid` 写入规则

```sail
function legalize_mwid(old : xlenbits, written : xlenbits) -> xlenbits = {
  if smwid_extract_locked(old)
  then old
  else {
    let new_wid = smwid_extract_wid(written);
    let new_locked = smwid_extract_locked(written);
    smwid_compose_value(new_wid, new_locked)
  }
}
```

如果旧值已经 locked：

- 返回 old
- 写入完全不起作用

如果还没 locked：

- 提取合法 WID
- 提取新的 lock bit
- 重新组成合法 `mwid`

---

## 写 `mwid` 时 helper 的调用顺序

```text
软件写 CSR 0x7C0
        |
        v
write_CSR(0x7C0, value)
        |
        v
legalize_mwid(old=mwid, written=value)
        |
        +-- smwid_extract_locked(old)
        |
        +-- 如果 old 未 locked：
              smwid_extract_wid(written)
              smwid_extract_locked(written)
              smwid_compose_value(new_wid, new_locked)
```

也就是说：

- 拆旧值，是为了判断之前是否已经 locked
- 拆写入值，是为了拿到软件想写的新 WID 和新 lock bit
- 重新组合，是为了清掉非法位，得到合法 `mwid`

---

## 写入例子：未锁定时

假设：

- `NWorlds = 4`，WID 字段宽度是 2 bit
- 旧 `mwid.L = 0`
- 软件写入的低位 WID 是 3
- 软件同时把最高位 lock bit 写成 1

执行结果：

```text
smwid_extract_locked(old) = false
smwid_extract_wid(written) = 3
smwid_extract_locked(written) = true
smwid_compose_value(3, true)
```

最终 `mwid` 变成：

```text
WID = 3, locked = true
```

## 代码：reset 时初始化 `mwid`

```sail
function reset_smwid() -> unit = {
  mwid = smwid_compose_value(sys_smwid_pmwid,
                             sys_smwid_locked_on_reset);
  csr_write_callback("mwid", mwid);
}
```

reset 时模型做两件事：

- 把 `mwid` 设置成平台默认 `pmwid`
- 根据配置决定是否一开始就 locked

`sys_control.sail` 中新增了 `reset_smwid()` 调用。

## 代码：注册 CSR

```sail
mapping clause csr_name_map = 0x7C0 <-> "mwid"

function clause is_CSR_accessible(0x7C0, priv, _) =
  currentlyEnabled(Ext_Smwid) & priv == Machine

function clause read_CSR(0x7C0) = mwid

function clause write_CSR(0x7C0, value) = {
  mwid = legalize_mwid(mwid, value);
  csr_write_callback("mwid", mwid);
  Ok(mwid)
}
```

注意：规范材料里 `mwid` CSR 地址仍是待定；当前实现使用 `0x7C0`。

## 关键词：物理地址和物理访问检查

CPU 执行 load/store 时，最终要访问物理地址。

访问物理地址前，模型会做几类检查：

- 地址是否对齐
- PMP 是否允许
- PMA 是否允许
- 现在还要检查 Smwid / World 权限

Smwid 检查发生在物理地址已经确定之后。

---

## 关键词：PMP 和 PMA

PMP 是 Physical Memory Protection。

它定义某些物理地址范围对不同特权级是否可读、可写、可执行。

PMA 是 Physical Memory Attribute。

它描述物理内存区域的属性，例如是否支持某类访问。

Smwid 不替代 PMP/PMA。

它是在这些机制之外再加一层“这个 WID 是否有权限”的检查。

---

## 代码：把 Smwid 接入物理检查

```sail
function phys_access_check forall 'n, 0 < 'n <= max_mem_access . (
  access : MemoryAccessType(mem_payload),
  pbmt : page_based_mem_type,
  priv : Privilege,
  paddr : physaddr,
  width : int('n),
  res_or_con : bool,
) -> option(ExceptionType) = {
  let smwidError : option(ExceptionType) =
    smwid_access_check(access, priv, paddr, width);
  let pmpError : option(ExceptionType) =
    pmpCheck(paddr, width, access, priv);
  let pmaError : option(ExceptionType) =
    pmaCheck(paddr, width, access, pbmt, res_or_con);

  match smwidError {
    Some(e) => Some(e),
    None() =>
      match (pmpError, pmaError) {
        (None(), None())     => None(),
        (Some(e), None())    => Some(e),
        (None(), Some(e))    => Some(e),
        (Some(e0), Some(e1)) =>
          Some(highestPriorityAlignmentOrAccessFault(e0, e1)),
      }
  }
}
```

注意：这里没有把 `wid` 作为参数传入 `smwid_access_check`。

`wid` 是在 `smwid_access_check` 内部根据 `access` 和 `priv` 取出来的。


```sail
function smwid_access_check(
  access : MemoryAccessType(mem_payload),
  priv : Privilege,
  paddr : physaddr,
  width : mem_access_width
) -> option(ExceptionType) = {
  let smwid_enabled = currentlyEnabled(Ext_Smwid);
  if not(smwid_access_controls_enabled(smwid_enabled, smwid_worlds_enabled))
  then return None();

  let wid = current_world_id_for_access(access, priv);
  if smwid_enabled & not(smwid_authorized(wid))
  then return smwid_unauthorized_access_result(access, wid, paddr);

  if not(smwid_worlds_enabled)
  then return None();

  match matching_smwid_world_region(paddr, width) {
    None() => smwid_unauthorized_access_result(access, wid, paddr),
    Some(region) =>
      if smwid_region_allows_access(region, access, wid)
      then None()
      else smwid_unauthorized_access_result(access, wid, paddr),
  }
}
```

也就是说，`phys_access_check` 只把“访问类型、特权级、物理地址、访问宽度”传进去。

`smwid_access_check` 自己根据当前模型状态计算 WID。

---

## 为什么不直接把 WID 传给 `smwid_access_check`

当前实现把“如何取得当前 WID”的逻辑集中在一个地方：

```sail
function current_world_id_for_access(
  _access : MemoryAccessType(mem_payload),
  _priv : Privilege
) -> xlenbits =
  if currentlyEnabled(Ext_Smwid)
  then get_mwid_wid()
  else smwid_extract_wid(sys_smwid_pmwid)
```

好处是：

- 没启用 Smwid 时，仍然使用平台默认 `pmwid`
- 启用 Smwid 时，从 `mwid` CSR 取 WID
- 将来支持 Smlwid/Sswid 时，可以在这个函数里按 privilege 选择不同 WID
- 调用方不用到处重复“当前 WID 怎么来”的规则

---

## 代码：WID 如何参与权限判断

```sail
let wid = current_world_id_for_access(access, priv);

if smwid_enabled & not(smwid_authorized(wid))
then return smwid_unauthorized_access_result(access, wid, paddr);

match matching_smwid_world_region(paddr, width) {
  None() => smwid_unauthorized_access_result(access, wid, paddr),
  Some(region) =>
    if smwid_region_allows_access(region, access, wid)
    then None()
    else smwid_unauthorized_access_result(access, wid, paddr),
}
```

判断分两层：

- 先检查这个 WID 是否在 `pmwidlist` 授权范围内
- 再检查目标 physical region 是否允许这个 WID 做当前类型的访问

---

## 关键词：World region

World region 是模型配置中的一段物理地址范围。

它描述：

- 从哪个物理地址开始
- 覆盖多大范围
- 哪些 WID 可以读
- 哪些 WID 可以写
- 哪些 WID 可以取指
- 哪些 WID 可以做页表遍历

这不是 ISA CSR，而是平台配置。

---

## 代码：World region 结构

```sail
struct SmwidWorldRegion = {
  base         : bits(64),
  size         : bits(64),
  read_wids    : bits(64),
  write_wids   : bits(64),
  execute_wids : bits(64),
  ptw_wids     : bits(64),
}
```

`read_wids` 等字段都是 bit mask。

bit i 为 1 表示 WID i 有该权限。

---

## 为什么有 `ptw_wids`

PTW 是 Page Table Walk，页表遍历。

当虚拟地址需要翻译成物理地址时，硬件或模型会读取页表项 PTE。

这也是一次内存访问。

所以它也要带 WID，也要检查权限。

`ptw_wids` 单独存在，是为了表达：

- 某个 World 可以读普通数据
- 但不一定可以把同一段内存当页表使用

---

## 代码：读路径如何带上 WID

```sail
val mem_read_priv_meta :
  forall 'n, 0 < 'n <= max_mem_access .
  (MemoryAccessType(mem_payload), page_based_mem_type, Privilege,
   physaddr, int('n), bool, bool, bool, bool)
  -> MemoryOpResult((bits(8 * 'n), mem_meta))

function mem_read_priv_meta
  (access, pbmt, priv, paddr, width, aq, rl, res, meta) = {
  let wid = current_world_id_for_access(access, priv);
  let result : MemoryOpResult((bits(8 * 'n), mem_meta)) =
    checked_mem_read(access, pbmt, priv, wid, paddr,
                     width, aq, rl, res, meta);
  match result {
    Ok(value, _) =>
      mem_read_callback(to_str(access), wid, bits_of(paddr), width, value),
    Err(e) =>
      mem_exception_callback(wid, bits_of(paddr), exceptionType_bits(e)),
  };
  result
}
```

读成功和读异常都能在 callback 里看到 WID。

---

## 代码：写路径如何带上 WID

```sail
val mem_write_value_priv_meta :
  forall 'n, 0 < 'n <= max_mem_access .
  (physaddr, int('n), bits(8 * 'n), MemoryAccessType(mem_payload),
   page_based_mem_type, Privilege, mem_meta, bool, bool, bool)
  -> MemoryOpResult(bool)

function mem_write_value_priv_meta
  (paddr, width, value, access, pbmt, priv, meta, aq, rl, con) = {
  let wid = current_world_id_for_access(access, priv);
  let result =
    checked_mem_write(wid, paddr, width, value, access,
                      pbmt, priv, meta, aq, rl, con);
  match result {
    Ok(_) =>
      mem_write_callback(to_str(access), wid, bits_of(paddr), width, value),
    Err(e) =>
      mem_exception_callback(wid, bits_of(paddr), exceptionType_bits(e)),
  };
  result
}
```

这保证 store/AMO 和 load 一样都会携带 WID。

## 配置校验为什么重要

Worlds 的安全性依赖平台配置。

如果配置本身错了，模型可能会出现很难理解的问题。

所以当前分支新增校验：

- Smwid 启用时必须有 Zicsr
- `nworlds` 必须在 `[1, XLEN]`
- `pmwid` 必须小于 `nworlds`
- `pmwidlist` 必须包含 `pmwid`
- region 权限 mask 不能设置不存在的 WID
- region size 不能为 0

---

## 未来实现可以做的工作

当前只实现 Smwid。

还没有实现：

- Smlwid：让 M-mode 控制低特权级 WID
- Smlwidlist：限制低特权级可用 WID 列表
- Smwdeleg / Sswid：把部分 WID 分配能力委托给 S-mode

所以现在的模型是：

```text
一个 hart，一个当前 WID，所有特权级共用
```

---

## 这次代码改动可以按四条线理解

第一条：CSR 线

- 新增 `mwid`
- reset 初始化
- lock / WARL 写入规则

第二条：权限线

- 配置 world regions
- `smwid_access_check`
- 接入 `phys_access_check`

第三条：事务线

- RAM / MMIO / callback 都传 WID

第四条：翻译线

- `translateAddr` 传 WID
- TLB entry 和 lookup 纳入 WID
